# Feature 084 — GoTrue Control-Plane Auth + Multi-Tenant Orgs + Cloud RBAC

Spec: [specs/084-gotrue-control-plane-auth/spec.md](../../specs/084-gotrue-control-plane-auth/spec.md) ·
Plan: [plan.md](../../specs/084-gotrue-control-plane-auth/plan.md)

## What changed

Replaces the hand-rolled human auth (`sb_sid` Redis session + the `studio-gotrue.ts` JWT shim +
the bespoke `users` table) with a **real control-plane GoTrue** service, and adds **multi-tenant
organizations** with **Cloud-style RBAC** (Owner / Administrator / Developer / Read-only),
**member management + email invites**, and **email (SMTP)**. PATs + the OAuth 2.1 MCP server are
unchanged in wire protocol — they just resolve identity from `auth.users` now.

| Area | Before | After |
|---|---|---|
| Operator identity | `public.users` (argon2) | GoTrue `auth.users` (control-plane `auth` service) |
| Human credential | `sb_sid` cookie + studio-JWT shim | GoTrue access+refresh JWT (`/auth/v1/*`) |
| JWT secret | n/a | `HKDF(master key, 'supastack-gotrue-jwt-v1')` — no new secret |
| Tenant model | `org` singleton | `installation` (apex+backups) + multi-row `organizations` (20-char ref ids) |
| Roles | `admin` / `member` | `owner`/`administrator`/`developer`/`read_only` (numeric ids 1–4) |
| Project ownership | implicit single org | `supabase_instances.org_id → organizations` (org-scoped authz) |

## Deploy (greenfield cutover)

> **Greenfield**: existing operator accounts + PATs are invalidated. Recreate the operator via
> `/setup` and re-issue PATs.

1. **Derive the GoTrue JWT secret** and add it to `infra/.env`:
   ```sh
   MASTER_KEY=<hex|base64> node scripts/derive-gotrue-secret.mjs >> infra/.env
   ```
2. **(Optional) SMTP** — to enable invites + password reset, set in `infra/.env`:
   `GOTRUE_SMTP_HOST`, `GOTRUE_SMTP_PORT`, `GOTRUE_SMTP_USER`, `GOTRUE_SMTP_PASS`,
   `GOTRUE_SMTP_ADMIN_EMAIL`, and `GOTRUE_MAILER_AUTOCONFIRM=false`.
3. **Build + bring up** the new `auth` service and the rebuilt api/worker/web:
   ```sh
   docker compose -f infra/docker-compose.yml build api worker auth
   docker compose -f infra/docker-compose.yml up -d db auth api worker caddy
   ```
   Migration `0017` creates the `auth` schema; GoTrue runs its own migrations; migration `0018`
   performs the cutover (idempotent — safe to re-run).
4. **Caddy** routes `/auth/v1/*` → `auth:9999`; the config is regenerated by the api
   (`POST /internal/caddy/reload`).
5. **`/setup`** creates the first operator in GoTrue + the `installation` row + the first org +
   an owner membership, and mints a master PAT (shown once).

## Platform API surface (Studio)

Built/real: `/platform/profile`, `/platform/profile/permissions`,
`/platform/profile/access-tokens`, `/platform/organizations` (list/create),
`/platform/organizations/:slug` (read/rename/delete), `…/roles`, `…/members`,
`…/members/:gotrue_id` (role/remove), `…/members/invitations` (list/send/cancel/by-token/accept),
`…/projects` (paginated). All mutations gate via `authorizeOrg(req, action, slug)`.

Native (GoTrue): `/auth/v1/{token,logout,user,settings,recover,verify,otp,health}`.

## Deploy notes (greenfield cutover)

1. GoTrue needs a `postgres` role in the control DB (its migrations `GRANT ... TO postgres`):
   `CREATE ROLE postgres SUPERUSER LOGIN;` once.
2. GoTrue's external URL env is `API_EXTERNAL_URL` (NO `GOTRUE_` prefix); also set `GOTRUE_JWT_ISSUER`.
3. **Recreate the `studio` container and clear its build.** Studio's dashboard auth client URL is
   `NEXT_PUBLIC_GOTRUE_URL`, which Next.js **inlines at `next build` time**. Editing the compose
   file and running `docker compose up -d` does NOT recreate the container or rebuild, so the old
   `…/api/v1` value stays baked and sign-in 404s on `/api/v1/token`. Required on the host:
   `rm -rf "$STUDIO_SOURCE_DIR/apps/studio/.next" && docker compose up -d --force-recreate studio`.
   Verify the new build baked `…/auth/v1`: `grep -rho 'supaviser.dev/auth/v1' …/.next/static | head`.
