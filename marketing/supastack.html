Self-hosted Supabase. N projects per VM.

Supastack runs a single docker compose stack that provisions isolated Supabase projects on demand. Each project gets its own Postgres, Auth, Storage, Realtime, Edge Functions, and Studio — with full supabase CLI compatibility.

$ git clone https://github.com/kmhari/supastack && cd supastack && ./up

Overview

What you get

Supastack is a small control plane (Fastify API, BullMQ worker, React dashboard) plus an opinionated per-project Supabase template. Everything runs as containers under one docker compose graph on one machine you own.

N

N isolated projects, one host

Each project runs in its own network namespace with its own Postgres volume, secrets, and ports. ~30–50 projects fit on a 4 vCPU / 16 GB VM.

$

Your supabase CLI just works

Login via device-code, link a ref, push migrations, generate types, deploy functions. Same API surface as Cloud — pinned against api.supabase.com/api/v1-json.

⌬

Direct Postgres on :5432

Custom STARTTLS+SNI proxy gives every project a strict-TLS endpoint at db.<ref>.<apex>:5432. psql, sqlx, rustls — all happy.

≋

Multi-tenant pooler

Top-level Supavisor 2.7 at pooler.<apex>:6543 serves all projects. A daily reconciler catches and self-heals seven classes of drift.

⧉

MCP, hosted

OAuth 2.1 server + multi-project MCP at mcp.<apex>/mcp. Paste into Claude / Cursor / Zed — full execute_sql tool surface.

⚿

Secrets in vault, not env

User secrets live in per-project vault.secrets. Functions read them with a 5s TTL cache — no container restart on save.

CLI Compatibility

Same commands. Same flags. Different host.

The upstream supabase CLI talks to the /v1 Management API. Supastack implements that same API verbatim. Point SUPABASE_API_URL at your VM, run supabase login, and the CLI doesn't know it left Cloud.

~/projects/huntvox
# authenticate via PKCE device flow — no PAT to copy/paste $ supabase login open: https://<apex>/dashboard/cli/login?session_id=… verification code: 8a3f-c910 ✓ token saved # link to a remote project — no password prompt, ever $ supabase link --project-ref enzyxdtrbosu ✓ linked · login role rotated · ttl 5m # push migrations $ supabase db push applying: 20260512_add_rls_policies.sql ✓ 20260514_user_billing_index.sql ✓ # generate types $ supabase gen types typescript --linked > types.ts ✓ 47 tables · 12 enums # ad-hoc query — audited, statement timeout enforced $ supabase db query --linked "select count(*) from auth.users" count ------ 12481
Architecture

Three layers, one docker compose.

Edge handles TLS and routing. The control plane handles provisioning, lifecycle, and the dashboard. The data plane is duplicated per project — fully isolated networks, volumes, and secrets.

L0 · EDGEtls + routing
caddywildcard *.<apex>L4 STARTTLS+SNI proxyHTTP-01 per-project certs Routes <ref>.<apex> to kong, db.<ref>.<apex>:5432 to per-project Postgres, pooler.<apex>:6543 to Supavisor.
L1 · CONTROLsingleton · 1 of 1
fastify apibullmq workerreact spapostgres 16redis 7supavisor 2.7 Speaks /api/v1 to the dashboard and /v1 to the upstream CLI. Provisions, monitors, rotates, repairs.
L2 · PROJECTn of n · isolated
dbauthrestrealtimestoragemetafunctionsanalyticsvectorimgproxystudiokong One full stack per provisioned project. Namespaced selfbase-<ref>-*. Add or remove in ~38 seconds.

Get started

One VM. Docker. A DNS record.

Any Linux host with Docker Engine ≥ 24 and a wildcard DNS record. The setup wizard does the TLS DNS-01 dance once. Everything after that is just provisioning more projects.

ubuntu@your-vm
# 1 — clone and configure $ git clone https://github.com/kmhari/supastack /opt/supastack $ cd /opt/supastack && cp infra/.env.example infra/.env $ $EDITOR infra/.env # set APEX, MASTER_KEY, admin email # 2 — bring it up $ sudo docker compose -f infra/docker-compose.yml up -d # 3 — finish the TLS DNS-01 dance in the setup wizard $ open https://<apex>/setup # 4 — provision your first project $ open https://<apex>/dashboard/new
