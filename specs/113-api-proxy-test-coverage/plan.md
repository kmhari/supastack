# Implementation Plan: Comprehensive API & Proxy Test Coverage

**Branch**: `113-api-proxy-test-coverage` | **Date**: 2026-06-08 | **Spec**: [spec.md](spec.md)

## Summary

Pure test addition. Four independently deliverable suites: (1) a fake-upstream integration test using the `TEST_KONG_BASE_URL` seam to assert exact proxy path/body/header contracts without any live infrastructure; (2) unit tests for the three management routes that currently have zero tests (`gen-types`, `migrations`, `postgrest-config`); (3) VM-backed smoke tests under `tests/integration/` that exercise the live proxy end-to-end against `supaviser.dev`; (4) black-box Fastify inject tests for the ~10 real-implementation platform-misc routes that currently have zero tests. No source files are modified, no migrations, no RBAC changes, no `/v1/*` contract drift.

## Architecture

### US2 — Fake-Upstream Tests (`platform-proxy-fake-upstream.test.ts`)

**Pattern**: `beforeAll` spins up a `http.createServer` bound to port 0; `afterAll` closes it. The server stores the last received request (`lastRequest`). Before each test, `process.env.TEST_KONG_BASE_URL` is set to `http://localhost:<port>`. After each test it is deleted. The test makes an HTTP inject call via Fastify, then asserts on `lastRequest`.

**What is mocked**: `resolveInstance` from `platform-proxy-helpers.js` returns a fake `{ portKong: 0, serviceRoleKey: 'test-srk', apiKey: 'test-key', ... }`. The fake upstream server records path/method/headers/body. `proxyToKong` is NOT mocked — it must call through to the fake upstream so the path rewriting logic is exercised end-to-end.

**Coverage per proxy group**:
- `pg-meta`: `GET /platform/pg-meta/:ref/tables` → upstream receives `GET /pg/tables`, `apikey` + `Authorization: Bearer` headers present
- `storage/list`: `POST /platform/storage/:ref/buckets/test/objects/list` → upstream receives `POST /storage/v1/object/list/test`, body has `prefix` field
- `storage/bucket-create`: `POST /platform/storage/:ref/buckets` with `{ id: 'my-bucket', type: 'private' }` → upstream body has `name: 'my-bucket'`
- `auth admin`: `GET /platform/auth/:ref/users` → upstream receives `GET /auth/v1/admin/users`, `apikey` + `Authorization: Bearer` headers present
- `analytics`: `GET /platform/projects/:ref/analytics/endpoints/logs.all` → upstream path is `/analytics/v1/api/endpoints/logs.all` (not doubled)
- `realtime` (via platform-misc): covered by existing `realtime-config.test.ts` unit test

### US3 — Management Route Unit Tests

**gen-types** (`gen-types.test.ts`):
- Mocks: `gen-types-service.js` (the `generateTypes` export), `project-store.js` (`getProjectByRef`)
- Uses same `buildApp` pattern as realtime-config/pgbouncer-config tests
- Tests: 200 happy path, 400 schema_not_found, 404 project not found, 401 unauthenticated, 409 instance not running, 502 upstream unreachable

**migrations** (`migrations.test.ts`):
- Mocks: `migrations-service.js` (`listMigrations`, `upsertMigration`, `deleteMigration`), error classes from `per-instance-pg.js` (`InstanceNotFoundError`, `InstanceNotRunningError`, `PerInstancePgConnectError`), `project-store.js`
- Tests: GET list 200, POST upsert 200, DELETE 200, POST 400 bad version format, DELETE 400 bad version, 404 (project not found via getProjectByRef), 409 instance not running, 401 unauthenticated

**postgrest-config** (`postgrest-config.test.ts`):
- Mocks: `project-store.js`, `runtime-config-store.js` (`getConfig`, `patchConfig`), `@supastack/db` (mock `db()` chain to return empty encryptedSecrets), `@supastack/crypto` (`decryptJson`, `loadMasterKey`)
- Tests: GET 200 with defaults (no stored config, no secrets), GET 200 with stored config, PATCH 200 valid body, PATCH 409 project not running, 404, 401
- Note: No Zod strict schema to test for PATCH — `postgrest-config` passes raw body to `patchConfig` which handles validation in the service layer; schema validation is at the service level not route level

### US1 — VM Smoke Tests (`platform-proxy-smoke.test.ts`)

**Location**: `tests/integration/` (existing vitest project, already collected)

**Guard pattern**: `const ENABLED = !!(process.env.TEST_API_URL && process.env.TEST_TOKEN_ADMIN && process.env.TEST_INSTANCE_REF)` then `describe.skipIf(!ENABLED)(...)`. Failed-project test additionally guarded by `TEST_FAILED_INSTANCE_REF`.

**HTTP client**: Node.js built-in `fetch` (Node 20). No new deps.

**Test matrix**:
| Test | Request | Assert |
|------|---------|--------|
| pg-meta tables | `GET /platform/pg-meta/:ref/tables` | 2xx, body is array |
| storage buckets | `GET /platform/storage/:ref/buckets` | 2xx, body is array |
| storage list | `POST /platform/storage/:ref/buckets/<first-bucket>/objects/list` | 2xx (not 400) |
| auth users | `GET /platform/auth/:ref/users` | 2xx |
| analytics logs | `GET /platform/projects/:ref/analytics/endpoints/logs.all` | not 500 |
| failed project | `GET /platform/pg-meta/:FAILED_REF/tables` | not 503 (2xx or 404/409) |

**Env vars**:
- `TEST_API_URL` — base URL e.g. `https://api.supaviser.dev`
- `TEST_TOKEN_ADMIN` — admin PAT
- `TEST_INSTANCE_REF` — healthy project ref
- `TEST_FAILED_INSTANCE_REF` — (optional) project ref in failed/crashed state

### US4 — Platform-Misc Black-Box Tests

**Pattern**: Identical to existing `platform-organizations.test.ts` and `platform-profile.test.ts` — Fastify inject through the full `platformMiscRoutes` plugin, db() mocked via `vi.mock('@supastack/db', ...)` returning a configurable fluent chain, `drizzle-orm` operators mocked as identity functions.

**Shared mock setup** (same across all three US4 test files):
```
vi.mock('drizzle-orm', () => ({ and: () => ({}), desc: () => ({}), eq: () => ({}), isNull: () => ({}) }))
vi.mock('@supastack/db', () => ({ db: () => dbChainMock, schema: { ... } }))
vi.mock('@supastack/crypto', () => ({ decryptJson: () => ({}), loadMasterKey: () => Buffer.alloc(32), generateRef: () => 'abcdefghijklmnopqrst' }))
```

The `dbChainMock` exposes a configurable `selectResult` variable so each test can control what `select().from().where().limit()` returns.

**`platform-projects.test.ts`**:
- Routes: `GET /platform/projects/:ref`, `PATCH /platform/projects/:ref`, `GET /platform/projects/:ref/databases`
- Mock `selectResult` to return a fake instance row for happy paths; `[]` for 404 paths
- Tests: GET 200 (shape has `ref`/`name`/`status`), GET 404, GET 401; PATCH 200 with name update, PATCH 200 empty body no-op, PATCH 404, PATCH 401; databases GET 200 shape, databases GET 404, databases GET 401

**`platform-access-tokens.test.ts`**:
- Routes: `GET /platform/profile/access-tokens`, `POST /platform/profile/access-tokens`, `DELETE /platform/profile/access-tokens/:id`, `GET /platform/profile/access-tokens/:id`
- Mock `mintApiToken` from `api-tokens-store.js` for POST
- Tests: GET list 200 (array), GET list 401; POST creates token 201 (has `token` field shown once), POST 401; DELETE 204 own token, DELETE 204 foreign token (404-as-204 pattern), DELETE 401; GET single 200, GET single 404, GET single 401

**`platform-misc-routes.test.ts`** (notifications + available-versions + auth hooks):
- Notifications routes return static shapes (`[]`, `{ unread: 0 }`, 204) — just need auth guard tests
- Available versions: static array — auth guard + shape
- Auth hooks (`GET/PATCH /platform/auth/:ref/config/hooks`): these use `app.inject()` internally to call `/v1/projects/:ref/config/auth` — the test must also register the auth-config routes (or mock `app.inject`). **Simpler approach**: the `platformMiscRoutes` plugin uses `app.inject` on itself, so we register both `platformMiscRoutes` + `authConfigRoutes` in the test app, then mock the auth-config backing stores (`runtime-config-store.js`). This tests the full delegation chain.

**`platform-misc-routes.test.ts` auth-hooks detail**:
- Register `platformMiscRoutes` + `authConfigRoutes` (management route) in one test Fastify app
- Mock `project-store.js` + `runtime-config-store.js` (same mocks as `realtime-config.test.ts`)
- `GET /platform/auth/:ref/config/hooks` → asserts 200 with hook-related fields in response
- `PATCH /platform/auth/:ref/config/hooks` → asserts 200 with updated hook fields
- Both routes: 404 when project not found, 401 when unauthenticated
