# Data Model вҖ” 010 Secrets management

**Date**: 2026-05-25

## Per-project Postgres (single source of truth)

Owned by `supabase_vault` upstream вҖ” selfbase does not define or migrate these. Listed for reference.

### `vault.secrets`

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | Generated by `vault.create_secret`. |
| `name` | `text` UNIQUE | User-chosen secret name (e.g., `OPENAI_API_KEY`). Reserved names rejected by api before insert. |
| `description` | `text` | Optional; selfbase leaves it empty (no UI field for it in v1). |
| `secret` | `text` | pgsodium-encrypted ciphertext. Never read directly вҖ” always go through `decrypted_secrets` view. |
| `key_id` | `uuid` | pgsodium key reference. Non-null after `pgsodium.create_root_key()` ran. |
| `created_at`, `updated_at` | `timestamptz` | Maintained by vault triggers. |

### `vault.decrypted_secrets` (view)

Read-only view that decrypts on `SELECT`. Used by both the api (digest computation + dashboard list) and the per-project edge runtime (TTL-cached injection map).

| Column | Type | Notes |
|---|---|---|
| `id`, `name`, `key_id`, `created_at`, `updated_at` | (same as above) | |
| `decrypted_secret` | `text` | Plaintext вҖ” only ever read inside the per-project Postgres process or by privileged callers. |

## Control-plane (selfbase DB)

### `instances` (existing table вҖ” `packages/db/schema.ts`)

**Added column** (additive, nullable per CLAUDE.md convention):

| Column | Type | Notes |
|---|---|---|
| `vault_enabled_at` | `timestamptz NULL` | Set by the `vault-enable` worker job (and by the provision pipeline directly) after a successful bootstrap + smoke test. Dashboard reads it to decide whether to surface the "Enable vault" button per project (visible only when NULL вҖ” re-enable is rare and only needed after a backup restore). |

Migration: `packages/db/migrations/0010_instances_vault_enabled_at.sql`

```sql
ALTER TABLE instances ADD COLUMN IF NOT EXISTS vault_enabled_at timestamptz NULL;
CREATE INDEX IF NOT EXISTS idx_instances_vault_enabled_at ON instances (vault_enabled_at) WHERE vault_enabled_at IS NULL;
```

(Partial index supports dashboard "needs enable?" lookups efficiently; index stays cheap because the predicate matches only un-enabled rows.)

## In-memory entities (no persistence)

### Runtime cache entry (`main/index.ts`, per functions container)

```ts
{
  ts: number,                      // ms since epoch of last successful refresh
  envVars: Record<string, string>  // name вҶ’ decrypted value, reserved names already filtered out
} | null
```

Default TTL window: **5000 ms** (`SELFBASE_VAULT_TTL_MS` env var override). Single shared in-flight refresh promise (request coalescing) prevents thundering-herd on TTL expiry.

### Reserved-secret descriptor (`packages/shared/src/reserved-secrets.ts`)

```ts
export type ReservedSecret = {
  name: string;          // e.g., 'SUPABASE_URL'
  description: string;   // one-line UI tooltip
};
export const RESERVED_SECRETS: ReservedSecret[] = [
  { name: 'SUPABASE_URL', description: 'Project API base URL вҖ” injected by selfbase' },
  // ... ~25 entries
];
export const RESERVED_SECRET_NAMES: Set<string> =
  new Set(RESERVED_SECRETS.map(s => s.name));
```

Materialized to `infra/supabase-template/volumes/functions/main/reserved-secrets.json` at api/worker build time (research.md Decision 4).

## Validation rules (cross-cutting)

| Rule | Enforced at |
|---|---|
| `name` is non-empty, matches `^[A-Za-z_][A-Za-z0-9_]*$`, length вүӨ 64 | api (`POST /v1/.../secrets`) вҖ” same regex as today's `secret-store.ts` |
| `name` is not in `RESERVED_SECRET_NAMES` | api at write time (rejected with 400) AND runtime at injection time (silently filtered, FR-014) |
| `value` is non-empty (Save form blocks; api also rejects) | web form + api |
| Batch upserts are atomic вҖ” partial-batch rejects fail the whole transaction | api (single `vault.*` transaction) |
| Vault extension presence | worker `vault-enable-job` smoke test + boot scan |
