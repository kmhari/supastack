# Implementation Plan: Migrate UI Primitives to Supabase Design System

**Branch**: `084-migrate-ui-primitives` | **Date**: 2026-06-02 | **Spec**: [spec.md](spec.md)

## Summary

Replace supastack's 18 locally-maintained shadcn UI primitives with vendored copies from Supabase's `packages/ui` component library, and migrate the CSS token system from shadcn conventions to Supabase's semantic token system. The result is a dashboard that uses the same design system as Supabase Studio вЂ” the custom DS Button (`type=` variants, loading/icon slots) and Supabase-tokenized versions of all other shadcn primitives.

**Approach**: Vendor source files (copy into tree). `packages/ui` is an unpublished internal workspace package вЂ” no npm distribution path exists. Vendoring is the same model shadcn itself uses and gives full control over the token bridge layer.

**Sequencing constraint**: CSS token migration (US1) must land before component vendoring (US2вЂ“US3), because the vendored components reference Supabase token names that won't resolve until the `@theme {}` block is updated.

## Technical Context

**Language/Version**: TypeScript + React 18, Vite 5, Tailwind CSS v4

**Primary Dependencies**: `radix-ui`, `class-variance-authority`, `lucide-react`, `clsx`, `tailwind-merge` вЂ” all already in `apps/web/package.json`; the vendored components use the same deps

**Storage**: N/A вЂ” frontend-only migration

**Testing**: Vitest (unit), Playwright (e2e) вЂ” `pnpm --filter @supastack/web test` and `test:e2e`

**Target Platform**: Browser (dashboard SPA at `apps/web/`)

**Project Type**: Frontend component migration вЂ” no new backend endpoints, no DB migrations

**Constraints**: Import path contract `@/components/ui/<name>` must remain stable; Playwright suite must pass; TypeScript build must pass with zero new errors

**Scale/Scope**: 18 component files replaced, ~136 call sites across `apps/web/src/`, 1 CSS token rewrite, 2 excluded components updated

**Source commit**: Pin to `$(cat infra/supabase-template/COMMIT)` вЂ” keeps dashboard primitives in sync with the Studio image running in per-project containers

## Implementation Phases

### US1 вЂ” CSS Token System (prerequisite)

**Files**: `apps/web/src/index.css`
**Effort**: Medium вЂ” token-by-token rewrite with mapping table from data-model.md
**Risk**: Low вЂ” all changes are CSS variable values; nothing structural changes
**Done when**: `pnpm --filter @supastack/web build` passes; dashboard renders with correct dark theme; Tailwind classes like `bg-brand-400`, `text-foreground-muted` resolve

### US2 вЂ” Button Migration

**Files**: `apps/web/src/components/ui/button.tsx`, `button.module.css`, `constants.ts`, `CopyButton.tsx`, 33 call-site files
**Effort**: Medium-high вЂ” 33 call sites need mechanical prop renames (`variant=` в†’ `type=`, size names)
**Risk**: Medium вЂ” all call sites must be updated; any missed site causes a TypeScript error (good: compiler catches misses)
**Done when**: TypeScript passes; all button-bearing pages render; Playwright button interaction tests pass

### US3 вЂ” Shadcn Component Bulk Migration

**Files**: 17 component files in `apps/web/src/components/ui/`
**Effort**: Low вЂ” mostly copy + fix relative imports; prop APIs are compatible
**Risk**: Low вЂ” shadcn-to-shadcn replacement; only token names change (handled by US1)
**Done when**: TypeScript passes; no call sites need updating (additive changes only)
