# Implementation Plan: Operator Admin Ops Console (read-only) + Setup Docs

**Branch**: `116-admin-ops-console` | **Date**: 2026-06-10 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/116-admin-ops-console/spec.md`

## Summary

Deliver the **read-only** operator console + public setup docs across four slices (Foundation, Resources, Queues, Cert/DNS), excluding the mutating Actions slice. Technical shape:

- **Frontend (`apps/web`)** gains a shared shell (router + layout + public nav), public `/docs/{index,cli,mcp}` (personalized from the live apex, reusing the existing `cli-wrapper.ts` + `apexApi`), and admin-gated `/admin/*` pages (Fleet, Project detail, Resources, Queues, Cert/DNS, System, Logs). Admin access reuses the existing GoTrue dashboard session via `auth-context` + a server-side role check.
- **API (`apps/api`)** gains read-only admin endpoints under `/api/v1/admin/*`, each gated by a new installation-admin RBAC action. The api holds **no** new privilege: it reads from DB tables + Redis (BullMQ) + the existing per-instance log proxy.
- **Worker (`apps/worker`)** gains one repeatable "observer" job that uses its existing docker socket + host mounts to sample per-project/host **resource usage**, **control-plane container health**, and **recent control-plane log tails**, writing them to two new tables. This confines all docker/host access to the worker (Constitution V).
- **DB** adds two idempotent, additive migrations: `resource_samples` (time-series) + `control_plane_snapshots` (latest-per-container).
- **Caddy** routes `/docs*` and `/admin*` ‚Üí `web:80` (boot Caddyfile + runtime `caddy-config.ts`), before the setup-gate catch-all.

No `/v1/*` Management API surface changes (Constitution IV untouched).

## Technical Context

**Language/Version**: TypeScript (Node 20). React 18 + Vite (`apps/web`); Fastify (`apps/api`); BullMQ worker (`apps/worker`); Drizzle + raw SQL migrations (`packages/db`).

**Primary Dependencies**: existing only ‚Äî React Router, the vendored Supabase design-system primitives (feature 084), axios client (`apps/web/src/lib/api.ts`), `@supastack/shared` (RBAC, QUEUES), BullMQ, Drizzle. **No new runtime dependency.**

**Storage**: control-plane Postgres (`db`). New tables `resource_samples`, `control_plane_snapshots`. Reads existing `supabase_instances`, `wildcard_certs`, `pg_edge_certs`, `audit_log`, backups store, plus Redis (BullMQ queue state).

**Testing**: Vitest unit/contract (api, shared, web, docker-control) + the RBAC matrix contract test. Live-VM smoke on supaviser.dev.

**Target Platform**: single VM (`supaviser.dev`), control-plane Docker compose stack.

**Project Type**: web (frontend `apps/web` + backend `apps/api` + worker `apps/worker`).

**Performance Goals**: console screens load in <2s with ‚â•50 projects (SC-008); observer tick completes well within its interval; per-project log query reuses the proven proxy.

**Constraints**: api gains **no** docker socket (privilege stays in the worker). Control-plane resources/health/logs are ~60s fresh (worker tick cadence); per-project logs are on-demand fresh. Migrations idempotent + additive; identity surrogate via `GENERATED BY DEFAULT AS IDENTITY` (not `CREATE SEQUENCE ‚Ä¶ OWNED BY` ‚Äî known migrate.ts owner gotcha).

**Scale/Scope**: dozens of projects (each ~11 containers) on one host; `docker stats --no-stream` over all containers per tick; bounded log tails (last N lines/container); resource-sample retention ~7 days.

## Constitution Check

| Principle | Applies? | How this plan complies |
|---|---|---|
| **I. Idempotent, additive schema** | YES | `resource_samples` + `control_plane_snapshots` migrations use `CREATE TABLE IF NOT EXISTS`, a `GENERATED BY DEFAULT AS IDENTITY` surrogate (avoids the `SEQUENCE ‚Ä¶ OWNED BY` owner crash), additive only. |
| **II. Secrets stay encrypted** | YES | The console never reads/decrypts instance secrets. Two leak vectors are explicitly redacted: failed-job payloads (FR-022) and control-plane log tails (api logs) ‚Äî known secret-bearing fields masked before persistence/display. |
| **III. Authorize every privileged action** | YES | New actions added to `packages/shared/src/rbac.ts` `ACTIONS` + granted to `owner`/`administrator` only; every `/api/v1/admin/*` endpoint gates via `app.authorize(req, <action>)`; the RBAC matrix contract test covers the new cells. |
| **IV. Supabase compat is a pinned contract** | YES (untouched) | All new endpoints live under `/api/v1/admin/*` (dashboard namespace). No `/v1/*` Management API path/shape changes ‚Üí zero upstream drift. |
| **V. Worker owns per-instance/host state** | YES | All docker/host access (stats, inspect, logs, disk) lives in the worker observer job. The new queue name is added to the shared `QUEUES` constant (`@supastack/shared`) ‚Äî no producer/consumer literal drift. |
| **VI. Spec-driven, evidence-based** | YES | Pure-function tests (snippet personalization, redaction, capacity math, role gate), a fake-docker collector test, the RBAC matrix test, and a live-VM smoke. Both happy + sad paths. |

**Gate result: PASS.** One conscious tradeoff tracked in Complexity Tracking (control-plane data staleness vs. api privilege).

## Complexity Tracking

| Decision | Why needed | Tradeoff / why simpler rejected |
|---|---|---|
| Worker "observer" job samples control-plane health + log tails to a table (instead of the api reading the docker socket on demand) | Control-plane health/logs need docker access; the api has none, and giving it the socket = root-equivalent privilege next to the master key | Per-project logs stay on-demand fresh (proxy); control-plane data is ~60s stale. Accepted: the spec scopes logs to "recent, not live tail," and confining docker to the worker upholds Constitution V + II far better than expanding api privilege. |
| Two new tables (time-series resources + latest-snapshot control-plane) | Resources need history/trends (FR-018); control-plane health/logs need only the latest | A single table would mix shapes/retention. Two narrowly-shaped tables keep queries + pruning simple. |
| New `admin.*` RBAC actions rather than reusing existing org actions | Installation-wide read is a distinct privilege from org-scoped project actions | Reusing org actions would conflate scopes and weaken the matrix's auditability. |
