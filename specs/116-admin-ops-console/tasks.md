## Dependencies & order

- **Setup (P1)** → **Foundational (P2)** → user stories.
- **US1 (docs)** depends only on Setup (shell/router/caddy) — NOT on Foundational auth/observer. Can ship first/standalone.
- **US2** depends on Foundational (T004 migration, T006 observer for system/logs, T008 guard, T009 client, T010 RBAC).
- **US3** depends on Foundational (T004 + T006 observer for `resource_samples`).
- **US4** depends on Foundational auth/RBAC/client (T008–T010) + redactor (T005) — NOT the observer.
- **US5** depends on Foundational auth/RBAC/client (T008–T010) — NOT the observer.
- Story order by priority: US1, US2 (P1) → US3, US4 (P2) → US5 (P3). US3/US4/US5 are independent of each other.

## Parallel execution examples

- **Setup**: T001, T002, T003 in parallel (distinct files).
- **Foundational**: after T004, run T005/T007/T008/T009/T010/T011 in parallel; T006 (observer) after T005 (redactor) + T004 (tables).
- **US1**: T012–T017 mostly parallel (distinct files); T018 (router wiring) last.
- **US2**: endpoints T019–T022 parallel; tests T023 parallel; pages T024–T026 parallel; T027 (router) last.
- **US4 + US5** can be built in parallel with each other and with US3 once Foundational auth/RBAC is done.

## Implementation strategy (MVP-first)

1. **MVP-A (ship docs)**: Setup (T001–T003 docs parts) + US1 → public CLI/MCP docs live. Zero auth, low risk.
2. **MVP-B (observe the fleet)**: Foundational + US2 → admin logs in, sees fleet/health/system/logs.
3. **Incremental**: US3 (resources) → US4 (queues) → US5 (certs), each independently demoable on the established shell.
