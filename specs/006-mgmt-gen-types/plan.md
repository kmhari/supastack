# Implementation Plan: Supabase CLI — gen types + migration + snippets + backups list/restore

**Branch**: `006-cli-mgmt-tier1` | **Date**: 2026-05-23 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/006-mgmt-gen-types/spec.md`

## Summary

Replace the `501 not_implemented` response for four Supabase CLI surfaces (gen types, migration *, snippets list/download, backups list/restore) by adding 11 new endpoints under the existing `/v1/*` Management-API mount in `apps/api`. Reuse existing per-instance Postgres access channels from feature 005 (pooler/direct), the existing PAT auth + RBAC from feature 003, and the existing backup job pipeline. Add one new control-plane entity (`restore_jobs`) + one new BullMQ job (`restore`) for the only architecturally novel piece — the async backup-restore worker.

## Technical Context

**Language/Version**: TypeScript 5.x / Node.js 20 (existing stack)

**Primary Dependencies**: Fastify 5 (existing), Drizzle ORM (existing), BullMQ (existing), `pg` 8 (existing for direct Postgres access), `undici` 6 (existing for outbound HTTP)

**Storage**: Per-instance Postgres for read-side data (gen-types introspection, `supabase_migrations.schema_migrations`, `user_content.content`). Control-plane Postgres for new `restore_jobs` table.

**Testing**: vitest (existing); CLI E2E shell scripts under `tests/cli-e2e/` (existing pattern from feature 003+005)

**Target Platform**: Linux server (Docker) — same control-plane stack as previous features

**Project Type**: web-service (api + worker + per-instance Supabase stacks)

**Performance Goals**:
- gen-types: <10s for ≤100 tables, <30s for ≤1000 tables (SC-001)
- migration list: <5s for ≤500 rows (SC-004)
- migration fetch: <30s for ≤500 rows (SC-004)
- snippets list: <2s for ≤200 snippets (SC-006)
- snippets download: <3s for ≤5MB body (SC-006)
- backup restore: <5min for ≤1GB DB (SC-007)

**Constraints**:
- Byte-compatible with Cloud Management API shapes (per https://api.supabase.com/api/v1-json)
- Zero regressions against existing P0 + feature-005 CLI commands (SC-010)
- Restore must roll back cleanly on partial failure — no zombie state (SC-008)

**Scale/Scope**: 4 user stories, 11 new endpoints, 1 new control-plane entity, 1 new BullMQ job. ~5-7 dev days estimated.

**Structure Decision**: Layer the new endpoints under the existing `apps/api/src/routes/management/` mount — same pattern as feature 003's `functions.ts` and `secrets.ts`. Per-instance Postgres access goes through a NEW shared `per-instance-pg.ts` helper that all four services use (decrypts secrets, opens a `pg.Client` to `host.docker.internal:<port_db_direct>`, returns a typed connection). Restore is the only piece that needs a worker job — everything else is synchronous read-or-write.

## Complexity Tracking

No constitution violations. The restore-job design (async + state machine + rollback guarantee) is justified by the asynchronous nature of the underlying ops (stop/swap/restart can take minutes) — synchronous request handling would time out and leave the project in an unknown state.

| Decision | Why | Simpler alternative rejected because |
|---|---|---|
| New `restore_jobs` control-plane table | Track async restore lifecycle, enforce one-in-flight-per-project | Returning a synthetic in-memory job id would lose state across api restarts and prevent dashboard surfacing |
| Pre-restore data-dir snapshot for rollback | FR-021 requires clean rollback on partial failure | Best-effort cleanup would leave projects in zombie state on worker crash |
| New shared `per-instance-pg.ts` helper | 4 services need typed per-instance PG access | Inlining the pattern in each service would duplicate secret decryption + connection lifecycle |
| Snippet visibility enforced in API, not via PG RLS | Studio's `user_content` lacks RLS in the current Supabase stack | Adding RLS to per-instance Studio schemas would diverge from upstream and break Studio's own queries |
