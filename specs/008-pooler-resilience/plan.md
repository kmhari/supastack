# Implementation Plan: Pooler resilience — reconciler + dashboard + PG password drift recovery

**Branch**: `008-pooler-resilience` | **Date**: 2026-05-24 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/008-pooler-resilience/spec.md`

## Summary

Build three layers of defense around the feature-005 pooler:
- **US1** Daily reconciler (BullMQ cron `0 3 * * *`) detects + auto-recovers tenant drift across `supabase_instances` × `pooler_tenants` × supavisor.
- **US2** Settings → Database dashboard panel surfaces health + per-project status + recent events + per-row re-register button.
- **US3** PG password drift: provision-time auth probe prevents shipping broken instances; reconciler classifies auth-class failures as `pg_password_drift`; admin reset endpoint runs `ALTER USER` via 127.0.0.1 trust auth.

Everything reuses existing tables (`pooler_tenants` gets a new status value + `last_reconciled_at` column; `pooler_events` gets new event types; `audit_log` gets new event types). One new entity: `reconciler_runs` for cron lifecycle tracking. One new dashboard route (`/settings/database`). All other code is additive to existing services.

## Technical Context

**Language/Version**: TypeScript 5 / Node.js 20 (existing stack)
**Primary Dependencies**: Fastify 5, Drizzle ORM, BullMQ, `pg` 8 (added in feature 006), `undici` 6 (existing for outbound HTTP to supavisor), React 18 / shadcn/ui / Tailwind (existing dashboard)
**Storage**: Control-plane Postgres for new `reconciler_runs` + additive columns. Per-instance Postgres reached via `per-instance-pg.ts` helper (active drift probe) + docker exec (ALTER reset path).
**Testing**: vitest unit; E2E shell scripts in `tests/cli-e2e/`
**Target Platform**: Linux server (Docker compose)
**Project Type**: web-service (api + worker + dashboard)
**Performance Goals**:
- Reconciler no-op tick: <2s for ≤50 projects (SC-001)
- Manual reconciler trigger → drift recovery: <5s end-to-end (SC-003)
- Dashboard panel first paint: <1s for ≤50 projects (SC-004)
- Reset endpoint → reconciler verify round-trip: <5s (SC-006)
- Provision-time drift detection: <10s after db healthy (SC-009)
**Constraints**:
- No new external dependencies
- Reconciler silent when nothing drifts (SC-007): no events for consistent path
- Per-instance failures isolated (FR-007)
- Single in-flight reconciler run (FR-002 — partial unique index)
- Provision auth probe MUST be retry-resilient (3× with 2s delay)
**Scale/Scope**: 3 user stories, 19 FRs, ~12-15 implementation tasks. ~1-1.5 days backend + ~1 day frontend.

## Complexity Tracking

| Decision | Why | Simpler alternative rejected because |
|---|---|---|
| Active 127.0.0.1 probe to classify `pg_password_drift` | Reliable classification (supavisor's error format unstable) | String-matching supavisor errors would misclassify on format changes |
| Provision-time auth probe with 3× retry | PG healthcheck reports ready ~1s before auth layer fully initialized | Single-shot probe would flake on race conditions |
| Synchronous single-instance reconciler pass on reset | Operator wants immediate green/red feedback in UI poll interval | Async + "check back later" forces 24h waits or extra button |
| BullMQ job over in-process cron | Matches existing cert-check + health-reconciler patterns | In-process setInterval loses persistence + observability |
