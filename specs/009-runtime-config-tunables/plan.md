# Implementation Plan: Runtime config tunables (postgres-config + auth-config)

**Branch**: `009-runtime-config-tunables` | **Date**: 2026-05-25 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/009-runtime-config-tunables/spec.md`

## Summary

Replace the catch-all `501 not_implemented` response for four Management API surfaces (`GET`/`PATCH /v1/projects/<ref>/postgrest`, `GET`/`PATCH /v1/projects/<ref>/config/auth`) by adding two new route modules under `apps/api/src/routes/management/`. Reuse the per-instance `.env` editor + `restartOrRollback` pattern from feature 003's secrets endpoint (`apps/api/src/services/secret-store.ts`). Persist config as a single JSONB snapshot per (project, surface) for fast GET — env reload remains the source of truth for container behavior. Add four RBAC actions mirroring upstream FGA permissions (`data_api_config.read/write`, `auth_config.read/write`). Validation Zod schemas match the upstream `api.supabase.com/api/v1-json` bounds byte-for-byte. Secret-typed auth fields are redacted on GET and treated as "leave unchanged" when the `***` sentinel appears on PATCH. One new control-plane table (`project_config_snapshots`) holds the post-merge persisted state.

## Technical Context

**Language/Version**: TypeScript 5.x / Node.js 20 (existing stack)

**Primary Dependencies**: Fastify 5, Drizzle ORM, Zod (all existing). No new runtime deps. The OpenAPI schema at `https://api.supabase.com/api/v1-json` is consumed at *spec/plan time* to derive bounds and field lists — not at runtime.

**Storage**: One new control-plane table `project_config_snapshots` (one row per `(instance_ref, surface)` where surface ∈ `'postgrest' | 'auth'`). Per-instance `.env` remains the source of truth for the container; the snapshot table holds the post-merge JSON that GET returns and that the OAuth-redacted PATCH round-trip merges against. Audit entries written to existing `audit_log` table.

**Testing**: vitest (unit) for `.env` upsert/remove edge cases, redaction sentinel logic, validation rejection cases, and OpenAPI-derived bounds. CLI E2E shell scripts under `tests/cli-e2e/` for the live-VM contract.

**Target Platform**: Linux server (Docker) — selfbase control-plane stack.

**Project Type**: web-service (api + worker + per-instance Supabase stacks)

**Performance Goals**:
- GET: <100ms p95 (single DB read, no container hit unless cache miss → one `.env` read).
- PATCH: <30s p95 end-to-end including container reload (SC-001, SC-002).

**Constraints**:
- Byte-compatible with Cloud Management API request/response shapes per the upstream OpenAPI.
- Zero regressions on prior CLI surfaces (gen-types, secrets, functions, login/link) — SC-008.
- Failed-restart MUST roll back `.env` and persisted snapshot atomically — FR-007 + SC-006.
- Plaintext secrets MUST NOT appear in GET responses — FR-003.

**Scale/Scope**: 2 user stories, 4 endpoints, 1 new control-plane table, 4 new RBAC actions, ~2 dev days estimated. No new BullMQ jobs (synchronous PATCH).
