# Implementation Plan: MCP Post-Ship Hardening (Feature 016)

**Branch**: `016-mcp-post-ship` | **Date**: 2026-05-26 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/016-mcp-post-ship/spec.md`

## Summary

Four follow-up improvements to feature 014 (Hosted MCP + OAuth 2.1):

1. **US1** — Set `statement_timeout = 8000` (8s) in the provision pipeline so new projects are query-safe by default.
2. **US2** — Filter deferred MCP tools (`create_project`, `get_cost`, `confirm_cost`, `get_security_advisors`, `get_performance_advisors`, `get_storage_config`, `update_storage_config`) from `tools/list` via a post-construction `setRequestHandler` override in `apps/mcp/src/server.ts`.
3. **US3** — Auto-uncomment the `analytics-v1-api` Kong routes in the supabase-template (new projects) + a one-shot worker boot job that patches all existing projects' kong.yml and restarts their Kong containers.
4. **US4** — Write 14 missing route-level tests across `oauth-authorize.test.ts` (7 cases) and `oauth-token.test.ts` (7 cases).

- All test mocks follow the `vi.hoisted + vi.mock` pattern from `oauth-register.test.ts`.

**Scale/Scope**: Single-operator deployment; no multi-tenancy concerns for these changes.

### Phase 2: US1 — statement_timeout at provision

**New file**: `apps/worker/src/services/pg-provision-defaults.ts`

```ts
import type { Client } from 'pg';

export async function applyProvisionDefaults(client: Client): Promise<void> {
  // 8s statement timeout — protects execute_sql / db query paths from runaway
  // queries. Idempotent: ALTER DATABASE SET is a no-op if value unchanged.
  await client.query(`ALTER DATABASE postgres SET statement_timeout = 8000`);
}
```

**Modify** `apps/worker/src/jobs/provision.ts`:
- After `await handleVaultEnable({ ref, source: 'provision' })` (line ~187)
- Add:
  ```ts
  // Feature 016 US1 — apply provision-time Postgres defaults (statement_timeout=8s)
  const pgDefClient = new pg.Client({
    host: 'host.docker.internal',
    port: inst.portDbDirect ?? inst.portPostgres,
    user: 'supabase_admin',
    password: secrets.postgresPassword,
    database: 'postgres',
    ssl: false,

    statement_timeout: 10_000,

  try {
    await applyProvisionDefaults(pgDefClient);
  ```

### Phase 3: US2 — MCP tool filter

**Modify** `apps/mcp/src/server.ts`:

After `const server = createSupabaseMcpServer({ platform });`, add:

```ts
import { ListToolsRequestSchema } from '@modelcontextprotocol/sdk/types.js';

const DEFERRED_TOOLS = new Set([
  'create_project',
  'get_cost',
  'confirm_cost',
  'get_security_advisors',
  'get_performance_advisors',
  'get_storage_config',
  'update_storage_config',
]);

// After createSupabaseMcpServer returns:
const origListToolsHandler = (server as any)._requestHandlers?.get('tools/list');
if (origListToolsHandler) {
  server.setRequestHandler(ListToolsRequestSchema, async (req, extra) => {
    const result = await origListToolsHandler(req, extra);
    return {
      ...result,
      tools: (result.tools ?? []).filter(
        (t: { name: string }) => !DEFERRED_TOOLS.has(t.name),
      ),
    };
  });
}
```

`DEFERRED_TOOLS` and the override block are module-level; the override runs once per new session entry (inside the `else` branch where `createSupabaseMcpServer` is called).

**Existing MCP tests** (`apps/mcp/tests/`): add one test case asserting that `tools/list` does not include any DEFERRED_TOOLS name.

### Phase 5: US4 — OAuth route-level tests

**New file**: `apps/api/tests/unit/oauth-authorize.test.ts`

7 cases (per issue #53 + authorize.ts code review):
1. GET valid session → 200 HTML with client_name in body
2. GET no session → 302 to `/dashboard/login?next=...`
3. GET invalid client_id → 400 `invalid_client`
4. GET invalid redirect_uri → 400 `invalid_request`
5. GET `code_challenge_method=plain` → 400 `invalid_request` (OAuth 2.1 hardening — already in `validateParams`)
6. POST authorize decision → 302 to redirect_uri with `code=...&state=...`; audit `oauth.code.issued`
7. POST deny decision → 302 to redirect_uri with `error=access_denied`; audit `oauth.consent.denied`

**New file**: `apps/api/tests/unit/oauth-token.test.ts`

7 cases (per issue #53 + token.ts code review):
1. authorization_code happy path → 200 + `access_token` (JWT shape) + `refresh_token` + `expires_in=3600`
2. Code reuse → 400 `invalid_grant`
3. Wrong code_verifier → 400 `invalid_grant`
4. Wrong redirect_uri → 400 `invalid_grant`
5. Wrong client_id → 400 `invalid_grant`
6. refresh_token happy path → 200 + new JWT + new refresh token
7. Refresh token reuse → 400 `invalid_grant` + verify `rotateRefresh` was called (reuse-detection path)
