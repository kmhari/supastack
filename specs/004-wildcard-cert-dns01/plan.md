# Implementation Plan: Wildcard TLS Cert via DNS-01

**Branch**: `004-wildcard-cert-dns01` | **Date**: 2026-05-23 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/004-wildcard-cert-dns01/spec.md`

## Summary

Extend supastack's `/setup` wizard with a DNS-01 certificate step: the wizard displays TXT records the operator manually adds at their DNS registrar, validates propagation via public DNS resolvers, then uses `acme-client` (npm) to complete the ACME challenge with Let's Encrypt and issue a `*.<apex>` + `<apex>` wildcard certificate. Caddy loads the cert from a shared Docker volume via `tls.certificates.load_files` — no custom Caddy build required. Renewal is operator-initiated (dashboard alert 30 days before expiry). Automated Cloudflare API renewal deferred to issue #6.

Reference implementation: `open-frontend/apps/api/src/services/acme-manual.ts` + `open-frontend/apps/edge/src/reload.ts`.

## Technical Context

**Language/Version**: TypeScript 5.x (Node.js 20) — existing stack

**Primary Dependencies**:
- `acme-client` npm — ACME protocol client for DNS-01 challenge (new dep in `apps/api/package.json`)
- `drizzle-orm` + `pg` — existing DB layer; two new tables
- `@supastack/crypto` — existing `encryptJson`/`loadMasterKey` for account key + cert key encryption
- `bullmq` — existing job queue; new `cert-check` daily cron job
- Node.js `dns/promises.Resolver` — TXT record DNS verification (no new dep)

**No new containers. No custom Caddy image.** Stock `caddy:2.8-alpine` stays as-is.

**Storage**:
- PostgreSQL 16 — two new tables: `wildcard_certs`, `cert_renewal_events`; migration `0003_wildcard_cert.sql`
- New Docker volume `certs-data` (mounted in `api` + `caddy`)

**Testing**: Vitest (existing); unit tests mock `acme-client` and `dns/promises.Resolver`; integration test runs against ACME staging (env `ACME_DIRECTORY_URL`)

**Target Platform**: Linux server (Docker Compose stack)

**Performance Goals**: TXT verification response ≤ 500ms (DNS lookup); cert issuance after TXT confirmed ≤ 30 seconds

**Constraints**:
- Private keys (ACME account key + cert key) must always be encrypted at rest
- Cert files on the shared volume are the source of truth for Caddy; DB stores a backup copy
- `ACME_DIRECTORY_URL` env override allows ACME staging in dev

### 2. DB Migration (0003_wildcard_cert.sql — idempotent)

```sql
CREATE TABLE IF NOT EXISTS wildcard_certs (
  id                uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id            uuid        NOT NULL REFERENCES org(id) ON DELETE CASCADE,
  apex              text        NOT NULL,
  status            text        NOT NULL DEFAULT 'pending'
                                  CHECK (status IN ('pending','awaiting_dns','verifying','issued','failed','disabled')),
  account_email     text        NOT NULL,
  account_key_pem   bytea       NOT NULL,
  order_url         text,
  challenge_records jsonb       NOT NULL DEFAULT '[]',
  cert_pem          text,
  key_pem           bytea,
  not_before        timestamptz,
  not_after         timestamptz,
  renewal_due       boolean     NOT NULL DEFAULT false,
  last_error        text,
  issued_at         timestamptz,
  created_at        timestamptz NOT NULL DEFAULT now(),
  updated_at        timestamptz NOT NULL DEFAULT now(),
  created_by        uuid        REFERENCES users(id) ON DELETE SET NULL,
  updated_by        uuid        REFERENCES users(id) ON DELETE SET NULL
);
CREATE UNIQUE INDEX IF NOT EXISTS wildcard_certs_apex_unique ON wildcard_certs (apex);

CREATE TABLE IF NOT EXISTS cert_renewal_events (
  id              uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  cert_id         uuid        REFERENCES wildcard_certs(id) ON DELETE CASCADE,
  org_id          uuid        NOT NULL REFERENCES org(id) ON DELETE CASCADE,
  triggered_by    text        NOT NULL CHECK (triggered_by IN ('initial','manual')),
  outcome         text        NOT NULL CHECK (outcome IN ('success','failure','in_progress')),
  error_message   text,
  cert_not_after  timestamptz,
  started_at      timestamptz NOT NULL DEFAULT now(),
  finished_at     timestamptz
);
CREATE INDEX IF NOT EXISTS cert_renewal_events_cert_idx ON cert_renewal_events (cert_id, started_at DESC);
```

### 4. Caddy Config Update (apps/api/src/services/caddy-config.ts)

`buildCaddyConfig()` now queries `wildcard_certs` in addition to existing queries:

```ts
const certRow = await db()
  .select({ apex: wildcardCerts.apex, status: wildcardCerts.status })
  .from(schema.wildcardCerts)
  .where(eq(schema.wildcardCerts.status, 'issued'))
  .limit(1);
const wildcardCert = certRow[0] ?? null;
```

**When wildcard cert exists** — add to `tls` app:
```ts
if (wildcardCert) {
  tlsApp.certificates = {
    load_files: [{
      certificate: `${CERTS_DIR}/${wildcardCert.apex}/cert.pem`,
      key: `${CERTS_DIR}/${wildcardCert.apex}/key.pem`,
      tags: [`wildcard:${wildcardCert.apex}`],
    }],
  };
}
```

**When wildcard cert exists** — replace `httpsServer` object with `tls_connection_policies`:
```ts
const httpsServer = {
  listen: [':443'],
  automatic_https: { disable_redirects: true },
  ...(wildcardCert ? {
    tls_connection_policies: [
      {
        match: { sni: [wildcardCert.apex, `*.${wildcardCert.apex}`] },
        certificate_selection: { any_tag: [`wildcard:${wildcardCert.apex}`] },
      },
      {},  // catch-all for on-demand TLS
    ],
  } : {}),
  routes: httpsRoutes,
};
```

**Without wildcard**: only existing `automation` block; no `certificates` block; no `tls_connection_policies`. Zero behavioral change for existing deployments.

### 6. cert-check BullMQ Job (apps/api/src/services/cert-check.ts)

Daily cron (`0 2 * * *`):
```ts
// Find issued certs within 30 days of expiry that aren't already flagged
const rows = await db().select().from(schema.wildcardCerts)
  .where(and(
    eq(schema.wildcardCerts.status, 'issued'),
    eq(schema.wildcardCerts.renewalDue, false),
    lt(schema.wildcardCerts.notAfter, new Date(Date.now() + 30 * 24 * 60 * 60 * 1000))
  ));
for (const row of rows) {
  await db().update(schema.wildcardCerts)
    .set({ renewalDue: true, updatedAt: new Date() })
    .where(eq(schema.wildcardCerts.id, row.id));
  await db().insert(schema.auditLog).values({ action: 'tls.renewal_due', ... });
}
```

### 7. Setup Wizard Step 4 (apps/web/src/pages/Setup.tsx)

New step `step === 'wildcard-cert'` with sub-states:

```
sub === 'loading'   → POST /api/wildcard-certs/initiate → show TXT records

sub === 'error'     → error message + retry button + skip link
```

**`WildcardCertCard.tsx`** component:

- Live DNS status row per record (`⏳ Propagating...` / `✅ Found`)
- "Verify" button (enabled always; disabled while verifying)
- "Skip for now" link (advances wizard without wildcard)

## Backward Compatibility

- **Existing deployments**: `buildCaddyConfig()` queries `wildcard_certs` — finds nothing — emits the identical config as today. Zero behavioral change.
- **Existing per-subdomain ACME certs**: Caddy's `caddy-data` volume retains them. Without `tls_connection_policies`, Caddy continues serving per-subdomain certs as before. When wildcard is added, the SNI policy takes over for `*.apex` — no disruption, existing certs just become unused.
- **`GET /org`**: Adding `hasCert` is a backwards-compatible addition (new field, no removals).
