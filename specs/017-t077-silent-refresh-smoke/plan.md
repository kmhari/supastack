## Implementation Design

### Script structure (`tests/cli-e2e/t077-silent-refresh.sh`)

The script extends the `oauth-dance.sh` pattern. Steps 1–4 are identical (DCR, PKCE, authorize, initial token exchange). New logic begins after token issuance:

```
Step 1   DCR register → client_id
Step 2   PKCE + state generation
Step 3   POST /v1/oauth/authorize (consent) → authorization code
Step 4   POST /v1/oauth/token (authorization_code) → access_token + refresh_token + expires_in
Step 5   GET /v1/profile with access_token → assert 200  (baseline: token valid at issuance)
Step 6   sleep(expires_in + 60)                          (access token genuinely expires)
Step 7   GET /v1/profile with access_token → assert 401  (negative-path gate, FR-003)
Step 8   POST /v1/oauth/token (refresh_token) → new access_token + rotated refresh_token
Step 9   GET /v1/profile with new access_token → assert 200  (SC-003 confirmed)
Step 10  Print [T077] PASS with timestamps → exit 0
```

Each step emits: `[T077] STEP: <name> | STATUS: <code> | ELAPSED: <s>s`

### Timing output

After step 4: `ISSUED_AT=$(date -u +%Y-%m-%dT%H:%M:%SZ)` and `RUN_START=$(date +%s)`.
After step 9: `REFRESHED_AT` computed; `TOTAL_ELAPSED=$(($(date +%s) - RUN_START))`.

Final PASS line:
```
[T077] PASS: SC-003 validated | total_elapsed: Xs | issued_at: <ISO8601> | refreshed_at: <ISO8601>
```

### Failure output

```
[T077] FAIL: <reason> | step: <step_name> | status: <http_code> | body: <truncated>
```
Then `exit 1`. Steps 1–5 failures = environmental/setup failure; steps 7–9 failures = contract regression.

### EXPIRES_IN handling

`EXPIRES_IN=$(echo "$TOKEN_RES" | jq -r '.expires_in')` — validated as positive integer before sleep. If invalid: `[T077] FAIL: unexpected expires_in value`.

### No reuse-detection step

This script does NOT re-use the old refresh token after rotation. Reuse-detection is already covered by `oauth-dance.sh` step 7. T077 validates only the happy-path contract.
